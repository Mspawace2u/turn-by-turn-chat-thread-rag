# How to Use This Guide

This guide is the working visual identity playbook for creating on-brand assets across product UI, landing pages, carousels, static social graphics, article visuals, and GlassHaus Field Notes.

Use it as the source of truth before making visual decisions.

The goal is not to “get inspired” and freestyle. The goal is to produce assets that feel recognizable, usable, premium, dopamine-safe, and aligned with the brand without requiring Patty to retrain or correct the same basics every time.

## Working Standard

Every asset should aim for **95%+ on-brand execution** before review.

That means the draft should already follow the guide’s rules for:

```txt
on-brand-output-check/
├── color logic
├── font choice
├── layout density
├── spacing
├── hierarchy
├── icon style
├── motion behavior when relevant
├── platform format
├── neurodivergent/burnout-aware readability
└── clear next-step or takeaway
```

Do not treat brand elements as decoration.

In this system, color, spacing, motion, typography, and layout all have jobs. They should reduce cognitive load, create signal, and help the audience understand what matters without feeling overstimulated.

## Before Creating Anything

Start by identifying the asset type.

```txt
asset-routing/
├── SPA / product UI
│   └── use product UI, layout, components, motion, and accessibility rules
├── landing page / offer page
│   └── use deck-on-scroll, CTA, hierarchy, and motion identity rules
├── carousel
│   └── use the carousel flow and one-idea-per-slide rule
├── single-post signal card
│   └── use one punchy idea with strong hierarchy and negative space
└── GlassHaus Field Note
    └── use the hand-drawn static explainer system
```

Do not mix asset systems unless the brief specifically calls for it.

A carousel is not a Field Note.
A Field Note is not a dashboard.
A dashboard is not a damn motivational poster.

## Default Design Priorities

When making tradeoffs, prioritize in this order:

```txt
priority-order/
├── 1. readability
├── 2. cognitive ease
├── 3. clear hierarchy
├── 4. brand recognition
├── 5. dopamine-safe visual interest
├── 6. platform fit
└── 7. creative flourish
```

Creative flourish comes last because the brand should feel alive, not chaotic.

## Decision Rules

Use the approved palette, font pairings, component rules, and motion rules unless the brief explicitly says otherwise.

Do not introduce:

```txt
do-not-introduce/
├── unapproved colors
├── random fonts
├── mixed icon libraries
├── red
├── generic SaaS blue
├── chalkboard texture
├── over-animated effects
├── crowded layouts
├── tiny text
└── Canva-template energy
```

When uncertain, simplify.

This brand gets stronger when the asset has fewer competing elements and a clearer point.

## Review Before Delivery

Before handing off any asset, check:

```txt
pre-delivery-check/
├── Can the main idea land in under 5 seconds?
├── Is there one obvious focal point?
├── Does the color usage mean something?
├── Is the text readable on mobile?
├── Is the layout spacious enough?
├── Does the asset avoid overstimulation?
├── Does it match the correct asset family?
├── Does it feel premium, human, and sharp?
└── Would someone recognize this as Patty’s after seeing 3–5 examples?
```

If the answer is no, revise before sending it to Patty.

## How to Ask for Clarification

Only ask when the guide and brief do not provide enough information to make a confident decision.

Ask one specific question at a time.

Do not ask Patty to restate known brand defaults.
Do not ask broad preference questions like “What vibe are you going for?”
The vibe is in this guide. Use it.

# 1. Core Visual Thesis

This brand system should feel like:

> **A premium dark-mode command center for founders whose brains are full, tabs are feral, and decision fatigue is running the damn meeting.**

The visual identity is not just “pretty dark UI.”  
It exists to reduce friction, create focus, and give small dopamine hits without turning the experience into a rave inside a spreadsheet.

The vibe:

```txt
GlassHaus brain relief
├── dark, calm, premium base
├── bright accents as signal, not decoration
├── generous negative space
├── one idea per screen
├── micro-interactions that confirm progress
├── clear CTA hierarchy
├── human-readable systems thinking
└── dopamine without sensory assault
```

The brand lane:

> **Premium dark-mode clarity for founders whose brains need signal, relief, and momentum.**

Stay there.  
Do not dilute this with beige SaaS templates, chaotic AI glitter, or “friendly startup blue.” That way lies Canva landfill.

# 2. Brand Design Principles

## 2.1 Focus First

The audience needs help with **directed focus, prioritizing, and decisioning**, so the visual system should avoid giant text walls, use short punchy copy, create visual breaks, and keep the user moving through one clear thought at a time.

**Design rule:**  
Every screen should answer:

> “Where do I look first, what matters most, and what do I do next?”

No mystery meat UI. No “choose your own adventure” nonsense unless choice is the actual point.

```txt
focus-first-design/
├── one dominant idea
├── one obvious next step
├── clear hierarchy
├── short copy blocks
├── visual breathing room
└── no clutter tax
```

## 2.2 Dopamine, Not Overstimulation

Dopamine should come from **short spurts of tension and release**, not constant stimulation.

Use dopamine through:

```txt
dopamine-safe-moments/
├── subtle glow on hover
├── small state changes after clicks
├── satisfying pill/button transformations
├── smooth scroll reveals
├── quick visual confirmation
├── progress cues
└── clean resolution moments
```

Avoid:

```txt
dopamine-flood-bullshit/
├── excessive animation
├── flashing effects
├── too many bright colors at once
├── competing CTAs
├── over-layered gradients
├── constant pulsing
└── motion that steals attention from the task
```

The goal is **reward without flooding**.

# 3. Core Color System

## 3.1 Required Base Palette

Every build should anchor on this core UI palette:

| Role | Color | Hex | Use |
|---|---:|---:|---|
| Main Background | Deep Black | `#050505` | Primary UI canvas |
| Primary Text | Off-White | `#F3F4F6` | Main readable text |
| Secondary Text | Muted Steel Blue-Gray | `#5B6B7F` | Subtitles, helper text, metadata |

**Rule:**  
The brights do **not** replace readability colors. They are signal, emphasis, borders, glow, feedback, and interaction.

## 3.2 Approved Brights Palette

These are the only approved bright colors:

| Color Name | Hex | Recommended Brand Role |
|---|---:|---|
| Electric Purple | `#9b5cff` | Primary action / selected state / main CTA |
| Totes Turquoise | `#2de2e6` | System logic / AI / workflow path / secondary accent |
| Punk Rock Pink | `#ff2f92` | Creative spark / margin note / playful emphasis |
| Screamer Green | `#3cff9e` | Success / resolved / “you did the thing” state |
| Highlighter Yellow | `#ffe44d` | Attention / caution / friction cue |
| Orange U Bright | `#f97316` | Escalation / bottleneck / “fix this shit” cue |

## 3.3 Recommended Cohesive Brand Combo

```txt
primary
└── Electric Purple #9b5cff

secondary
└── Totes Turquoise #2de2e6

support accents
├── Screamer Green #3cff9e
├── Highlighter Yellow #ffe44d
├── Orange U Bright #f97316
└── Punk Rock Pink #ff2f92 sparingly
```

**Why:**  
Electric Purple already functions as the primary action/selected-state signal in the visual system, so making it the main CTA color keeps the design language consistent.

# 4. Color Usage Rules

## 4.1 Electric Purple

Use for:

```txt
Electric Purple/
├── primary CTA
├── active menu state
├── selected pills
├── action icons
├── key borders
├── hover glow
└── high-value emphasis moments
```

Do **not** use it everywhere. If everything screams, nothing signals.

## 4.2 Totes Turquoise

Use for:

```txt
Totes Turquoise/
├── AI/system logic
├── workflow paths
├── bot/agent states
├── automation cues
├── secondary interactive cues
└── “the system is working” moments
```

This is the “calm tech intelligence” color.

## 4.3 Green / Yellow / Orange

Use these like a visual ops dashboard:

```txt
status-logic/
├── Green = resolved / win / safe to proceed
├── Yellow = attention / slow down / possible friction
└── Orange = bottleneck / broken handoff / needs action
```

This helps overwhelmed users make faster decisions without decoding five paragraphs of UX copy.

## 4.4 Punk Rock Pink

Use for:

```txt
Punk Rock Pink/
├── idea sparks
├── creator notes
├── margin notes
├── playful highlights
├── “human voice” accents
└── rare surprise-and-delight moments
```

Pink is seasoning. Not soup.

## 4.5 No-Red Rule

Do not use red in this brand system.

Orange handles escalation, bottlenecks, costly friction, and broken handoffs. Red is not needed and creates alarm-state intensity that does not fit the dopamine-safe, burnout-aware visual system.

```txt

├── H1: bold, tight, high contrast, max 2 lines
├── H2: bold, section-level thought
├── H3: smaller label-style header
├── Body: short, scannable, 1.2–1.3 line height
├── Microcopy: muted steel blue-gray
```

## 6.2 Copy Density Rule

No giant blocks. Ever.

Use:

```txt
short-copy-blocks/

├── 1 CTA per action zone
├── 1 primary instruction at a time
├── visual breaks before the brain taps out
└── enough breathing room to scan fast
```

## 6.3 Label + Button Text

Labels and buttons should use uppercase with tracking.

```txt
label-style/
├── uppercase
├── tracked lettering
├── short phrases
├── strong contrast
└── no vague corporate mush
```

Avoid CTAs like:

```txt
bad-cta-copy/

├── Submit
├── Discover
└── Optimize Your Journey
```

Banish “optimize your journey” to the LinkedIn Thought Leader Parking Lot.

```txt
motion-personality/
├── calm
├── precise
├── responsive
├── lightly playful
├── signal-driven
├── dopamine-safe
└── never chaotic
```

## 13.3 Motion Principles

```txt
motion-principles/
├── clarify the next step
├── confirm user action
├── reveal sequence
├── reduce uncertainty
├── create small reward moments
├── support scanning
└── stop before it becomes stimulation clutter
```

## 13.4 Core Motion Types

```txt
motion-types/
├── entrance motion
│   └── soft fade / slight rise
├── hover motion
│   └── glow / elevation / border brightening
├── selection motion
│   └── subtle pop or scale
├── progression motion

    └── clean fade or instant close when clarity matters
```

## 13.5 Emphasis Pulse

The **emphasis pulse** is a signature SPA motion pattern used to direct attention without adding noise.

```txt
emphasis-pulse/
├── starting opacity: 40%
├── ending opacity: 100%
├── purpose: spotlight important text
├── best for: key phrases, status words, tiny reveals, decision cues
├── avoid on: long paragraphs, dense instructions, legal/disclaimer text
└── use sparingly: one focal point at a time
```

Best uses:

```txt
use-emphasis-pulse-for/
├── “Start here”
├── “Your next move”
├── “Bottleneck found”
├── “High-friction step”
├── “AI can handle this”
└── “Human judgment needed”
```

## 13.6 Emphasis Pulse Guardrail

The pulse should feel like:

> “Hey, look here.”

Not:

> “WELCOME TO THE NEON FUNERAL PARADE.”

Keep it smooth, readable, and occasional.

```txt
├── section reveal: ~300–600ms
├── CTA transformation: long enough to register
└── no delayed feedback that makes users think the app broke
```

# 14. Neurodivergent / Burnout-Specific Design Rationale

## 14.1 Why Dark Mode Works Here

The dark base creates a calmer field of attention and lets bright accents act as intentional signals.

This supports prioritization because the user’s eye does not have to fight a white-background wall of chaos.

```txt
why-dark-mode-works/
├── lowers visual intensity
├── supports contrast
├── gives bright accents meaning
├── creates calm dimensionality
└── helps the user scan for signal
```

## 14.2 Why Bright Accents Work

Brights provide controlled reward and attention cues.

They create little “you’re here / this matters / that worked” moments without forcing the whole UI to sparkle like a Vegas slot machine with unresolved childhood issues.

```txt
bright-accent-purpose/
├── attention cue
├── action cue
├── selected state
├── friction signal
├── progress signal
└── reward moment
```

## 14.3 Why Pills Work

Pills feel contained, tappable, and low-threat.

They create a clear action boundary, which helps users understand what can be selected, changed, or advanced.

```txt
why-pills-work/
├── clear boundaries
├── easy tap targets
├── low cognitive load
├── friendly shape language
├── strong selected-state potential
└── good for micro-decisions
```

## 14.4 Why One Ask Works

One ask lowers the activation cost.

Decision fatigue eats momentum. A single micro-step helps the user move without requiring them to rebuild their entire executive function stack from scratch.

```txt
one-ask-rule/
├── one question
├── one action
├── one CTA
├── one decision
└── one next step
```

## 14.5 Why Motion Must Stay Subtle

Motion should confirm action, reveal sequence, and support orientation.

It should not become the task.

For burned-out users, too much motion becomes another input to process.

```txt
motion-for-overwhelm/
├── confirm
├── guide
├── reward
├── orient
└── stop before it distracts
```

# 16. Carousel System

Carousels should behave like a slow-burn thought path.

```txt
carousel-flow/
├── slide 1: hook / tension / pattern interrupt
├── slide 2: name the problem
├── slide 3: make the hidden friction visible
├── slide 4: reframe the issue
├── slide 5: show the better pattern
├── slide 6: give the micro-action
└── final slide: one CTA or save/share prompt
```

Each slide gets **one job**.

Not five. Not a dissertation. Not “here’s my entire nervous system in 12pt type.”

## 16.1 Visual Hierarchy for Social

Use the same base system:

```txt
social-visual-hierarchy/
├── dark base: #050505
├── readable text: #F3F4F6
├── secondary text: #5B6B7F
├── primary signal: Electric Purple #9b5cff
├── system/AI signal: Totes Turquoise #2de2e6
├── status accents: green / yellow / orange
└── playful accent: pink, sparingly
```

## 16.2 Carousel Text Rules

```txt
text-rules/
├── headline: big, punchy, max 1–2 lines
├── body: short, conversational, easy to skim
├── labels: uppercase + tracking
├── emphasis: color, scale, or spacing, not ALL THE THINGS
└── CTA: one clear action
```

Recommended body density:

```txt
per-slide-copy/
├── best: 8–20 words
├── acceptable: 20–35 words
└── danger zone: 40+ words unless it is a quote/testimonial card
```

## 16.3 Dopamine Without Overstimulation

Static graphics create dopamine through:

```txt
dopamine-safe-static-visuals/
├── satisfying color contrast
├── clear reveal from slide to slide
├── bold but controlled accent moments
├── visual rhythm
├── “oh shit, that’s me” copy moments
└── clean resolution at the end
```

Avoid:

```txt
static-graphic-nope-list/
├── every color on one slide
├── too many glow effects
├── tiny text blocks
├── chaotic background patterns
├── multiple CTAs
├── crowded diagrams
└── gradients that compete with the message
```

## 16.4 Social Graphic Component Rules


# 18. GlassHaus Field Notes: Static Explainer Graphics

**GlassHaus Field Notes** are static, feed-native visual explainers used for Substack article images, Substack Notes, X posts, LinkedIn posts, and standalone thought-leadership graphics.

They are **not carousels** and they are **not mini slide decks**.

They are one-page visual field notes that make a workflow, pattern, bottleneck, or AI/system concept easy to understand fast.

```txt
GlassHaus Field Notes/
├── format: single static image
├── orientation: portrait
├── aspect ratio: 8.5 x 11 paper ratio
├── use case: Substack, Notes, X, LinkedIn
├── visual style: clean dark glass field-note board
├── content style: hand-drawn but highly readable
└── purpose: explain one useful idea without dev-speak
```

## 18.1 Core Visual Thesis

The image should feel like:

> A smart operator mapped the messy workflow on a clean dark glass board so a non-dev founder can finally see what the hell is happening.

```txt
GlassHaus Field Notes/
├── premium dark systems surface
├── hand-drawn human explanation
├── clear agent/workflow logic
├── founder-friendly language
├── useful visual tension
└── one clean takeaway
```

## 18.2 Format Lock

```txt
format-rule/
├── default orientation
│   └── portrait
├── default aspect ratio
│   └── 8.5:11 paper ratio
│       approximately 0.77:1 width-to-height
├── best use case
│   └── Substack / feed-native explainers
└── do not default to
    ├── square carousel graphics
    ├── 16:9 slide layouts
    └── wide dashboard mockups
```

Future format specs can be created separately:

```txt
future-format-specs/
├── GlassHaus Field Notes: Portrait Feed Spec
├── GlassHaus Field Notes: Landscape Deck Spec
│   ├── 16:9
│   └── 5:4
└── GlassHaus Field Notes: Social Crop Spec
    ├── square
    └── vertical mobile
```

Do not make one style spec carry every format like an overworked group project mom.

## 18.3 Surface + Background

Use a **clean near-black glass surface**, anchored to the brand base.

```txt
surface-rules/
├── base: #050505
├── finish: clean black glass
├── texture: subtle dot-grid depth
├── edge detail: slight reflection/refraction
├── glare: never crosses important text
└── mood: premium ops lab, not classroom board
```

## 18.4 Hard No

```txt
field-notes-kill-list/
├── chalkboard texture
├── dusty residue
├── erased ghost marks
├── smudged marker trails
├── streaky dry-erase marks
├── cute classroom whiteboard
├── messy conspiracy wall
├── generic Canva infographic energy
├── sterile SaaS architecture diagram
├── neon cyberpunk overload
├── unreadable handwritten chaos
├── AI robot mascot circus
└── primary-color productivity poster
```

That distinction matters because image tools will absolutely hear “field notes” and try to hand you a 1998 math classroom if you leave the door cracked. Rude little chaos engines.

```txt
├── slightly imperfect
├── readable at mobile size
└── never childish or messy
```

```txt

├── sharp phrases
├── no tiny paragraph clusters
├── no over-explaining
├── one bottom-line takeaway
└── readable in-feed within 3–5 seconds
```

## 18.6 Field Notes Color Logic

Use the approved GlassHaus/PDub color system only.

```txt
field-notes-color-logic/
├── off-white: structure, default text, neutral arrows
├── turquoise: bot, agent, automation, system logic
├── purple: human judgment, HITL, approval, exception handling
├── green: win state, resolved flow, relief
├── yellow: cool friction, caution, slowdown, early attention point
```

**No red. No random Tailwind blue. No “but this looked cute” palette crimes.**

## 18.7 Connector Logic

```txt
├── yellow line

├── orange line
│   └── bottleneck, breakage, rework, or escalation
└── green line
    └── resolved path / clean handoff / win state
```

```txt
callout-use-cases/
```

## 19.2 Pop-Up Boxes

```txt
popup-box-rules/
├── mobile: max 80% of viewing area underneath
```
