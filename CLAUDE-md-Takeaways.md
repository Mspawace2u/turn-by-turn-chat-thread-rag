Rule 1: Make Claude think before it writes The first rule is simple: Claude should not silently assume. Before writing code, it should state what it believes the task means. If there are two possible meanings, it should show them. If something is unclear, it should stop and ask. This fixes one of the biggest AI coding mistakes. You say: Add export user data. Claude might assume: all users JSON format admin-only access all database fields no privacy filtering simple sync endpoint That is dangerous. A better Claude response should be: I need to confirm the export scope, format, fields, privacy limits, and expected data size before writing this. This one habit can save hours. Not because it makes Claude smarter, but because it stops Claude from being quietly wrong. Add this to your file: ## Think Before Coding Before implementing, state assumptions clearly. If the request has multiple possible meanings, list them before choosing. If something is unclear, ask before writing code. If a simpler approach exists, mention it before building the bigger one. Rule 2: Make the first solution boring The second rule is about simplicity. Claude loves to build flexible things. Sometimes too flexible. You ask for a discount function. It creates a discount strategy interface, three classes, a config layer, and future extension points nobody asked for. That may look professional, but it is not always good engineering. Good code solves the current problem cleanly. The viral CLAUDE.md puts this very directly: use the minimum code that solves the problem, avoid speculative features, avoid one-use abstractions, and simplify if the solution is far bigger than needed. Add this: ## Simplicity First Write the minimum code that solves the current problem. Do not add features that were not requested. Do not create abstractions for one-use code. Do not add configurability unless the project already needs it. If the solution feels bigger than the problem, simplify it. This is not anti-architecture. It is anti-fake architecture. There is a difference. Rule 3: Make Claude touch only the wound This is the rule I would keep even if I removed everything else. AI coding agents often make drive-by changes. They fix a bug, but also: rename variables reformat nearby code change comments move imports rewrite a helper clean up something unrelated change style because they prefer another style This is how small tasks become risky tasks. The surgical rule tells Claude to touch only the code needed for the current task. If it notices dead code, it can mention it. It should not delete it unless asked. Every changed line should connect back to the original request. Add this: ## Surgical Changes Only change files and lines directly required for the task. Do not refactor unrelated code. Do not reformat nearby code. Do not improve comments, names, or structure unless asked. Match the existing style, even if another style looks better. If you notice unrelated issues, mention them after the task. Do not fix them silently. This rule cuts the review burden. You stop asking: Why did it change this file? And start asking: Did this fix the thing I asked for? That is the right question. Rule 4: Make every task verifiable The fourth rule is the most useful for real coding. Do not tell Claude only what to do. Tell it what done means. Fix the bug is weak. Better: Reproduce the bug with a failing test, fix it, then run the test again. Add validation is weak. Better: Add tests for invalid input, make them pass, and confirm valid input still works. Refactor this is weak. Better: Refactor this without changing behavior, run existing tests before and after, and list the files changed. The repo calls this goal-driven execution.
